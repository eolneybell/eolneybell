Project 3
Bottleneck: A Browser-Based Coding Game
Bottleneck is a game that provides players with practice solving problems in Javascript. Indeed, designing this game also improved my skills in Javascript, HTML, and CSS. Coding was in some sense the focus of this project for me, leaving the aesthetics with some room for improvment. Nevertheless - wireframes, heuristic evaluations, and usability tests were still central to the development of this project.
bottleneck.world

An early review of the literature focused on gamification, usability heuristics, and teaching with games. In particular, I used a series of heuristics developed by Isbister & Schaffer (2008) to both guide game development and, later, to evaluate the game.

A sampling of heuristics.
Usability Test 1

Once a working version of the game had been developed, I recruited six participants for a first round of testing, which was intended to ascertain the usability of the game in its existing form at bottleneck.world. Participants would play in ‘Practice Mode’, without an opponent capable of winning (or doing much of anything) largely to test the intuitiveness of the game concept, the tutorial, and the strength of the user experience. The test protocol was designed to be open-ended, such that participants would come to their own conclusions without being influenced by my own biases. The script naturally included some probes for me to use if I desired more detail. Insights that came from this round of testing included the consequences of the users’ inability to modify the size of window’s in the game’s GUI, the fact that pointers given by the tutorial frequently appeared offscreen and were difficult to find, and the unfamiliar appearance of some of the game elements (e.g. resources, such as rocks), among others.

A screenshot from the initial functioning prototype of the game

These findings led me to create a series of mockups, which formed a redesigned prototype of the game. Among the modifications was an expanded HUD (heads up display) to address issues participants had distinguishing informational displays and buttons, the ability to modify the size of windows, a redesigned tutorial, and simpler & more realistic game elements (e.g. resources).

A sample of the new interface
Usability Test 2

My second set of usability tests consisted of nine participants, only three of whom were brand new. I split my six participants up from last time into two groups of three: one group would return to the version of the game that they had played before, only this time without the tutorial. The other three returning participants would test the new prototype but would also not use the tutorial. The only individuals exposed to the new tutorial were the three new participants.

The findings from the second round of usability tests produced a somewhat more compact user interface. This was partly a response to usability issues (e.g. trouble clicking on the game’s map) that arose from the enlarged HUD, as well as heavier reliance on (and stated desire for) tooltips. A mini map was added to aid in situation awareness (SA), which was found to be lacking for a number of participants.

A screenshot showing the latest iteration of the game interface

Ideally, I would have continued testing the game from here and provided a full summative evaluation. However, as the project was largely academic and limited in scope, I wrapped things up after the second round of usability tests and game enhancements.
